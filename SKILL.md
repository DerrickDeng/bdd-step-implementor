# BDD Step Implementor

This skill defines the attach workflow for implementing Cucumber BDD steps with playwright-cli on the live browser launched by Cucumber.

It is only valid when playwright-cli is attached to the same Chromium process that Cucumber launched. If the CLI is connected to a different browser process, the analysis is invalid and this skill must not be used.

Reference files mentioned below are relative to this skill directory. Script commands are shown as repo-root-relative paths.

## Key Principles

Five principles underpin the entire workflow. Re-check them whenever you are unsure how to proceed.

1. **BDD text is a frozen specification.** Step text is authored by BA/QA as a contract. Changing it silently breaks the contract even if tests still pass. The only permitted feature file edit is adding the isolation tag.

2. **MCP observes, stubs execute.** The attach session shares the live browser with Cucumber. Any MCP action (click, type, navigate) mutates state the stub expects to control, causing failures or misleading passes. Keep MCP read-only; let `impl.js` drive all interactions.

3. **One run owns the port.** Two concurrent runs on the CDP port produce interleaved protocol files and unpredictable browser state. Always stop the previous run before starting a new one, and scope all bridge files under a unique `runId`.

4. **Stop on proven impossibility.** If MCP confirms the expected content is absent on the correct page, that is a scenario/data error, not a locator problem. Report the case error immediately instead of burning retries.

## Phase 1: Discovery

1. Read the feature file in full. **The BDD step text is frozen.** In BDD workflows, the step text is authored by BA/QA as a specification ŌĆö it is not yours to edit. Even if you see typos or grammatical errors, preserve them exactly. Your job is to write code that makes the existing steps pass, not to improve the steps themselves. Never change scenario title, step text, or step order. The only permitted feature-file modification is adding the isolation tag.

   **Pay close attention to comments** (`#` or `##` lines above steps). Comments may contain **implementation directives** ŌĆö specific instructions for how the step should be implemented (e.g., store a value to World, validate a World variable, set up API interception). Treat these directives as requirements when writing `impl.js` in Phase 3D.

2. Detect scenario type:
   - `Scenario Outline` with `Examples`:
     - `scenario_type = "outline"`
     - `row_count` - number of data rows (excluding header)
     - `first_row_line` - **CRITICAL**: Line number of the first DATA row, which is the line AFTER the header row

       **How to find it correctly:**
       1. Locate `Examples:` keyword in the feature file
       2. Next line = header row with `|placeholder1|placeholder2|` ŌåÉ **SKIP THIS**
       3. Next line = first data row with actual values ŌåÉ **USE THIS LINE NUMBER**

       Example:
       ```
       Examples:              # Line 18
         |loginUser|optValue| # Line 19 ŌåÉ header (skip)
         |TW_USER  |111111  | # Line 20 ŌåÉ first data (use this)
       ```
       Result: `first_row_line = 20`

     - `placeholders` - list of `<placeholder>` names
   - Plain `Scenario`:
     - `scenario_type = "scenario"`

3. **Detect profile and run dry-run**:

   a. **Determine suggested profile**:
      - Read `profile.all_profiles` from `.claude/project-profile.json`
      - Determine `suggested_profile` using this algorithm:

        **Step 1: Check package.json scripts**
        - Look for default test/start scripts (e.g., `"test": "npm run sit"`)
        - If found, use that profile ŌåÆ DONE

        **Step 2: Extract and match Feature/Scenario tags**
        - Extract ALL `@tag` from both Feature line AND Scenario line
        - Find which tags match `all_profiles` entries (case-insensitive comparison)
        - Collect all matching profile names into `matching_profiles` list

        **Step 3: Handle matching profiles**
        - If `matching_profiles` is empty ŌåÆ go to Step 4
        - If `matching_profiles` contains exactly one profile ŌåÆ use it with reason "from @{TAG} tag" ŌåÆ DONE
        - If `matching_profiles` contains multiple profiles:
          - Check if `sit` or `SIT` is in `matching_profiles` ŌåÆ use `sit` with reason "from @SIT tag (preferred over @UAT)" ŌåÆ DONE
          - Otherwise, use `matching_profiles[0]` with reason "from @{TAG} tag" ŌåÆ DONE

        **Step 4: Default to SIT**
        - If `sit` or `SIT` exists in `all_profiles` ŌåÆ use it with reason "SIT default" ŌåÆ DONE

        **Step 5: First available**
        - Use `all_profiles[0]` with reason "first available" ŌåÆ DONE

      - Store: `suggested_profile`, `reason`, `all_profiles`

   b. **Load command templates** from `.claude/project-profile.json`:
      - Load from `profile.profiles[suggested_profile].commands`
      - Store as: `dry_run_template`, `run_template`, `attach_plain_template`, `attach_outline_template`
      - Store `active_profile = suggested_profile` in memory

   c. **Run dry-run** using `dry_run_template`:
      ```bash
      <dry_run_template with {feature_path} replaced>
      ```

4. Build the implementation queue:
   - `undefined` / `pending` => queue
   - `skipped` => already implemented, skip
   - `ambiguous` => stop and report

   Deduplicate template steps for Scenario Outline and assign step index `N = 1..k`.

5. Plan PO / step-def targets ŌĆö for each step, determine the Page Object and Step Definition file:
   - Glob using `page_object_glob` from the profile, find the best-fit PO by domain/page name. Read candidate POs in full to understand existing methods and locator strategy.
   - Derive the Step Definition file from the PO using the project's naming convention. If it exists, read it in full. If not, note it for creation.
   - Present the full mapping table and wait for user confirmation. `[existing]` = already implemented, `[pending]` = to implement.

   Plain Scenario:
   ```
   Implementation Plan:
   Step 1: "Given I am on the homepage"    ŌåÆ [existing] HomePage.ts / home.steps.ts
   Step 2: "When I click Media"            ŌåÆ [pending]  HomePage.ts / home.steps.ts
   Step 3: "Then I should see news list"   ŌåÆ [pending]  MediaPage.ts / media.steps.ts
   ```

   Scenario Outline (use template steps with `<placeholder>`, note row count):
   ```
   Implementation Plan ŌĆö Scenario Outline: Verify navigation (4 rows)
   Step 1: "When I click on the \"Media\" navigation dropdown"  ŌåÆ [pending] HomePage / home.steps
   Step 2: "Then I can see the dropdown list expanded"           ŌåÆ [pending] HomePage / home.steps
   Step 3: "When I click \"<optionType>\" option in dropdown"    ŌåÆ [pending] HomePage / home.steps  ŌåÉ parameterized
   Step 4: "Then I can see the \"<keyword>\""                    ŌåÆ [pending] MediaPage / media.steps ŌåÉ parameterized
   ```

6. Generate isolation tag:
   - `{snake_case_title}_{4-char-hex}`
   - confirm no collision before writing it:
     ```bash
     rg -n --fixed-strings "{isolation_tag}" . "$SKILL_DIR" mcp-bridge
     ```
   - if any hit appears, regenerate the suffix and check again
   - add the tag to the target scenario
   - save `feature_baseline_hash` immediately after the tag is written
   - always compute it with:
     ```bash
     node "$SKILL_DIR/scripts/feature-baseline-hash.js" <feature_file_path>
     ```
   - `feature_baseline_hash` is the MD5 of the feature file after adding the isolation tag and before any later validation
   - this saved value is the baseline you must carry through Phase 4; do not replace it with a later hash
   - do not use `shasum`, `shasum -a 256`, or any ad-hoc hash command here; the quality gate expects this exact MD5 value

7. Build final commands using the loaded templates (from step 3b):

   Collect the **scenario tags** ŌĆö all `@tag` annotations on the target Scenario / Scenario Outline line and its parent Feature line. Build a `{tag_filter}` that combines the isolation tag with every scenario tag using `and`:

   ```
   {tag_filter} = "@isolation_tag and @tag2 and @tag3 ..."
   ```

   For example, if the scenario has `@TW @UAT @Digi` and the isolation tag is `@my_scenario_a1b2`:
   ```
   {tag_filter} = "@my_scenario_a1b2 and @TW and @UAT and @Digi"
   ```

   Then build the actual commands by filling in the templates from the selected profile:

   - `run_command` = fill `{tag_filter}` into `run_template`
   - `attach_command`:
     - **Scenario Outline**: fill `{feature_path}`, `{line}` (= `first_row_line`), AND `{tag_filter}` into `attach_outline_template`
       - **CRITICAL**: All three parameters must be filled in
       - The `{line}` must be the first DATA row (the row after the header row with `|placeholder|` names)
       - The `tag_filter` ensures correct environment configuration (e.g., BASE_URL selection based on `@TW`, `@Digi` tags)
       - **Before setting the command, re-count the line number from the current feature file** (after adding the isolation tag) to ensure accuracy
       - Example: If Examples header is on line 18 and first data row is on line 19, use line 19
       - Example result: `npm run uat -- src/features/tw/explore/productDetails.feature:19 --tags "@my_scenario_a1b2 and @TW and @UAT and @Digi"`
     - **Plain Scenario**: fill `{tag_filter}` into `attach_plain_template`
   - `attach_command` is for the attach workflow in Phase 3. For Scenario Outline it intentionally runs only the first Examples row so MCP can pause on template steps.
   - `run_command` is for Phase 4 final validation. It runs the full isolated tagged scenario, including all Examples rows.

   **Why tag_filter matters**: Many Cucumber configs use `process.argv` to detect tags and set environment-specific variables (BASE_URL, API endpoints, etc.). Omitting scenario tags from the filter can cause "URL is invalid" or similar config errors even when the isolation tag and line number are correct.

   **Why profile matters**: Cucumber configs often use `process.argv` to detect `--profile=` flags and set environment-specific variables (BASE_URL, API endpoints, credentials). Wrong profile ŌåÆ wrong environment ŌåÆ "URL is invalid" errors.

## Phase 2: Generate Bridge Stubs

Build the steps JSON array:

```json
[
  {"type": "When", "text": "I click on the {string} dropdown", "index": 1},
  {"type": "Then", "text": "I can see the dropdown expanded", "index": 2}
]
```

Generate the temporary stub step-def file. The script reads defaults from `project-profile.json` automatically, but you can override with explicit flags:

```bash
node "$SKILL_DIR/scripts/generate-stubs.js" \
  <stub_step_def_file> \
  '<steps-json>' \
  --world-import <world_import> \
  --world-type <world_type>
```

Use the values from Phase 0. The script reads the profile for defaults, so if the profile is correct you only need to pass `<stub_step_def_file>` and `'<steps-json>'`.

Verify it compiles:

```bash
<tsc_check> <stub_step_def_file>
```

Error recovery:

- If `tsc` fails on imports or unresolved modules, check that `stub_step_def_file`, `world_import`, and `world_type` match the actual framework structure from Phase 0.
- If `generate-stubs.js` rejects the JSON input, inspect the step text for quoting or escaping mistakes and regenerate the JSON rather than editing the generated file manually.
- If the project requires a different TypeScript invocation to type-check a single file, follow the project convention, but keep the generated stub file temporary.

Timeout note:

- Do not try to set the run budget inside the generated stubs.
- `start-run.js` is the single source of truth for timeout budgeting.
- The generated stubs consume two launcher-owned values:
  - `MCP_STEP_RUN_TIMEOUT_MS` for the whole attach run deadline
  - `MCP_STEP_IMPL_TIMEOUT_MS` for a single `impl.js` execution attempt
- `MCP_STEP_TIMEOUT_MS` remains as a backward-compatible alias for the run timeout when older launchers invoke the stubs.

The generated stubs do this for each queued step:

1. Write `mcp-bridge/<runId>/step-N-pause.json`
2. Wait for `mcp-bridge/<runId>/step-N-impl.js`
3. Execute it on the real paused World
4. Write either `step-N-pass.signal` or `step-N-error.txt`
5. Retry after each rewritten impl

If the queue contains repeated step text with the same Cucumber expression, the generated stubs must register that expression only once and dispatch by occurrence order at runtime. Repeated bindings for the same expression make Cucumber ambiguous and are invalid.

The bridge protocol is run-scoped. Never use the flat `mcp-bridge/step-N-*` paths directly. Resolve the active `runId` from `mcp-bridge/latest-run.json`, then use the run-local directory and launcher manifest under `mcp-bridge/<runId>/`. For example, protocol files live at `mcp-bridge/<runId>/step-N-pause.json` and launcher-owned process metadata lives at `mcp-bridge/<runId>/launch.json`. This prevents stale runs from colliding with the active attach session.

## Phase 3: Run the Paused Test in Attach Mode

### Pre-flight: Validate attach_command

Before spawning the run, run a 5-second dry-run to catch command errors immediately ŌĆö this avoids waiting 120 s for `wait-for-step.js` to time out:

```bash
{attach_command} --dry-run 2>&1 | tail -30
```

**For Scenario Outline, verify your attach_command includes ALL required parts:**
- Ō£ģ Feature path: `src/features/...`
- Ō£ģ Correct line number (first data row, NOT header row)
- Ō£ģ Complete tag_filter with isolation tag AND all scenario tags

Example of a correct Scenario Outline attach_command:
```bash
npx cucumber-js --profile=uat src/features/tw/explore/productDetails.feature:19 --tags "@my_scenario_a1b2 and @TW and @UAT and @Digi"
```

Checks (all must pass before proceeding):

1. **Scenario count Ōēź 1** ŌĆö output must contain `1 scenario` or similar. If it shows `0 scenarios`, the tag filter or feature path is wrong.
2. **No compilation errors** ŌĆö no `Cannot find module`, `SyntaxError`, or TypeScript errors. If present, the stub file needs fixing.
3. **No config errors** ŌĆö no `Error: Profile "xxx" does not exist`. If present, the profile flag in the command is wrong.
4. **Correct environment message** ŌĆö for projects with tag-based config (e.g., "Using TW digi url in UAT"), verify the expected message appears. If you see "Url in UAT is invalid" or similar, the scenario tags are missing from tag_filter.

Common fixes:
- **0 scenarios / tag filter**: check quoting ŌĆö call the shell as `--tags "@iso_tag and @bdd"` so the launcher receives a clean argv value `@iso_tag and @bdd`. Do not pass nested literal quotes such as `--tags '"@iso_tag and @bdd"'`. Also check that feature-level tags (e.g. `@bdd`) are included alongside scenario tags.
- **0 scenarios / wrong line number**: for Scenario Outline, the line number MUST point to the first DATA row (after the header).
  **How to verify**: Re-read the feature file, find the `Examples:` section, skip the header row (the one with `|placeholder|` names), and use the line number of the NEXT row (the one with actual data values).
  **Example**: If the header `|loginUser|optValue|` is on line 18, and data `|TW_USER|111111|` is on line 19, use line 19 in the command.
- **Config error / missing scenario tags**: some projects parse `process.argv` for tags to set BASE_URL or other env variables. If the dry-run shows environment errors but scenario count is correct, add all scenario tags to the tag_filter.
- **0 scenarios / wrong profile**: verify that the `active_profile` you selected in Phase 1 step 2a exists in `profile.profiles`. The attach_command should use the correct profile's base_command (e.g., `npm run uat --` for uat profile).
- **Wrong line (Scenario Outline)**: **CRITICAL CHECK** - Open the feature file and manually count lines:
  1. Find `Examples:` keyword
  2. Next line = header row with `|placeholder|` ŌåÉ skip this
  3. Next line = first data row ŌåÉ THIS is the line number to use
  If you used the header line number by mistake, the command will fail with 0 scenarios. Always use the data row line number.
- **Cannot find module**: the stub file path or a require path inside it is wrong ŌĆö recheck `stub_step_def_file` from the profile

Do not call `start-run.js` until the dry-run shows the correct scenario count and no errors.

Start the isolated test through the run launcher. Do not background a raw shell command yourself.

```bash
node "$SKILL_DIR/scripts/start-run.js" \
  --port <cdp_port> \
  --log test-results/mcp-step.log \
  --step-count {queue_length} \
  --clean-port \
  -- {attach_command}
```

Use `cdp_port` from the profile (default 9222).

`start-run.js` is the only supported launcher for this workflow. It:

1. Ensures the CDP port is usable before launch
2. Assigns a unique `runId` and exports it to the stub runtime
3. Truncates `test-results/mcp-step.log` for the new run
4. Auto-restores previously validated `step-N-impl.js` files when the command matches
5. Stores the active run pointer in `mcp-bridge/latest-run.json` and the canonical launcher manifest in `mcp-bridge/<runId>/launch.json`
6. Computes the whole-run timeout budget from `--step-count` unless you explicitly override it
7. Launches a watchdog that terminates the detached test process when the run deadline expires

Do not write the run log into `mcp-bridge/`. Only protocol files belong there.

Timeout budgeting: pass `--step-count {queue_length}` and the launcher computes the budget (3 min + 45 s/step, cap 25 min). Override with `--timeout-ms` or `--impl-timeout-ms` if needed. Do not tune timeouts in `generate-stubs.js`; the launcher owns them.

### Progress display

```text
[1/3] WAITING  "When I click on the {string} dropdown"
[1/3] WRITING  "When I click on the {string} dropdown"
[1/3] PASS     "When I click on the {string} dropdown"
[2/3] WAITING  "Then I can see the dropdown expanded"
```

### For each step N

#### A. Wait for pause

```bash
node "$SKILL_DIR/scripts/wait-for-step.js" N --timeout-ms 120000
```

When `wait-for-step.js` returns, check the exit code:

| Exit Code | Meaning | Action |
|-----------|---------|--------|
| 0 | Success - step found | Continue to Phase 3B |
| 2 | Timeout - process was alive | See `references/wait-for-step-errors.md` ŌåÆ Exit 2 |
| 3 | Process died (confirmed) | Read log, diagnose crash, restart run |
| 4 | Run deadline expired | Check progress, restart with more time if needed |

**For Exit 2 (timeout), read `references/wait-for-step-errors.md` for detailed diagnostic steps.** The reference explains:
- How to double-check process liveness (race condition handling)
- Case A: No bridge files yet (stuck in setup/earlier steps)
- Case B: Paused on earlier step (implement that step first)
- Case C: Test progressing but slow (retry wait with extended timeout)

**For Exit 3 (process died):**
1. Inspect `mcp-bridge/<runId>/launch.json` for the canonical PID, timeout, and log metadata
2. Read log: `tail -100 test-results/mcp-step.log`
3. If the log is empty, inspect `mcp-bridge/<runId>/launch-failure.json` for the launcher-side diagnostic snapshot
4. Diagnose: missing dependency, syntax error, hooks failure, browser crash
5. Fix the issue
6. Restart: `stop-run.js --clean-port` then `start-run.js`

**Important:**
- DO NOT manually check process with `ps aux` or similar commands
- Trust the exit codes - they already encode process liveness
- Read `references/wait-for-step-errors.md` for complete troubleshooting steps

**Pause JSON structure** - When a step pauses, `step-N-pause.json` contains:

**Core identification:**
- `stepIndex` - The step number (N)
- `step` - The full step text (e.g., "When I click the {string} button")
- `type` - Step type ("Given" | "When" | "Then")
- `args` - Runtime arguments for this execution (captured from the actual test run)
- `url` - Current page URL when the step paused

**For repeated step text:**
- `stepKey` - The Cucumber expression pattern
- `occurrence` - Which occurrence of this pattern (1, 2, 3...)
- `occurrenceCount` - Total occurrences of this pattern in the scenario

**Timing/deadline info:**
- `runId` - Unique run identifier
- `runDeadlineAt` - Absolute timestamp when the whole run expires (ms since epoch)
- `runTimeoutMs` - Total run budget in milliseconds
- `implAttemptTimeoutMs` - Timeout for a single impl.js execution (default 60s)
- `deadlineAt` - Same as runDeadlineAt (legacy field)
- `timeoutMs` - Same as runTimeoutMs (legacy field)
- `timestamp` - When this pause was created (ms since epoch)

**Directory info:**
- `bridgeDir` - Full path to the active run's bridge directory
- `bridgeRoot` - Full path to the mcp-bridge root directory
- `runStartedAt` - ISO timestamp when the run started

**Important:** For repeated step text, always key your reasoning, filenames, and progress updates by `stepIndex`, not just step text. `step 3` and `step 7` are distinct work items even if the Cucumber expression (`stepKey`) is identical. Use `occurrence` to understand which repetition you're implementing.

#### B. Attach playwright-cli to the live browser and confirm identity

Use `playwright-cli` exclusively in this phase. The `playwright-test` MCP is not valid here ŌĆö it would launch a new Chromium process and cannot see the paused Cucumber session.

Attach once per run through the helper script:

```bash
node "$SKILL_DIR/scripts/attach-cli.js" --step N
```

The helper waits for `mcp-bridge/<runId>/cdp-ws.txt`, falls back to the live CDP `/json/version` endpoint if needed, closes any stale daemon for that run-scoped session, attaches `playwright-cli`, and verifies the paused step URL when `--step N` is passed. This prevents accidentally reusing a stale CLI session or attaching to an `about:blank` browser.

The helper prints the exact `CLI` value to use, for example:

```text
[attach-cli] CLI="playwright-cli -s=bdd-1a2b3c4d"
```

This spawns a persistent node daemon bound to that run's short session name. All subsequent `playwright-cli` invocations in Phase 3 must pass the same `-s=...` flag to reach the daemon.

For convenience, define a shell variable at the start of Phase 3:

```bash
CLI="playwright-cli -s=<session printed by attach-cli.js>"
```

Then:
- `$CLI tab-list` ŌĆö list pages on the attached browser
- `$CLI snapshot --filename=.playwright-cli/step-<N>.yml` ŌĆö capture a11y tree to a file. **Always pass `--filename`.** Without it, the CLI prints the full YAML inline to stdout and consumes a lot of tokens; with it, you get back only a file path you can `Read` with `offset`/`limit`. Use `step-<N>.yml` for the primary snapshot per step; if you need another during the same step (e.g. after a state change), add a short suffix like `step-<N>-after-click.yml`.

- `$CLI screenshot` ŌĆö take a screenshot

#### C. Analyse the paused page with playwright-cli

Use playwright-cli interactively on the exact paused page. Read `references/cli-snapshot-rules.md` for the full snapshot reuse protocol.

**playwright-cli is observation-only in this workflow.** The attach session shares the exact browser the test is running in. Any CLI action (click, type, hover, navigate) would mutate the page state that the stub expects to control, causing the test to fail or produce misleading results. Use `$CLI snapshot` plus read-only `$CLI eval` only.

Key constraints and their reasons:

- **`$CLI eval` must remain read-only** ŌĆö DOM mutations (clicking, typing, dispatching events) corrupt the paused page state that `impl.js` will execute against.
- **Do not use `browser_generate_locator`** ŌĆö it produces CSS selectors that are fragile to layout changes. Hand-writing locators from snapshot evidence (role, name, testid) produces more maintainable tests.
- **After taking a snapshot, always run a test-id scan via `$CLI eval`** ŌĆö `data-testid` attributes are not visible in the accessibility snapshot, so a DOM query is needed to discover them. Use the discovery query from `references/cli-snapshot-rules.md`. Prefer `getByTestId()` when a matching test-id is found.
- **Do not use `$CLI eval` for other DOM hunting** ŌĆö the snapshot already exposes roles, names, and text. Only use additional `$CLI eval` calls when the snapshot genuinely lacks the information you need.
- **Do not use `$CLI eval` to rediscover** text, visibility, or dropdown contents already visible in the current snapshot ŌĆö it wastes context and retries on information you already have.

**MANDATORY: Before analyzing the snapshot, read these reference files in full:**

- `references/code-patterns.md` ŌĆö **Read the entire file**, including locator priority order, coding conventions, forbidden patterns, and all project-specific patterns observed in Phase 0
- `references/cli-snapshot-rules.md` ŌĆö for fallback DOM query examples and snapshot reuse discipline

**Verify you understand both the locator priority AND the coding patterns before choosing any approach.** Using incorrect patterns (wrong locator priority, wrong page access method, wrong assertion style) violates project standards and will cause implementation failures.

**Use already-passed steps as context.** When writing `impl.js` for step N, read the validated `impl.js` files from earlier steps (in `mcp-bridge/<runId>/`). They reveal what the page actually contains, how users interact with it, and what gets validated ŌĆö giving you a richer understanding of the page than the snapshot alone. Combined with the BDD step text (which describes business intent), this lets you understand both the full case narrative and each step's role within it.

For parameterized steps, use actual `args` from pause JSON to validate the current row, but write generic code in `impl.js`. If the snapshot/DOM evidence yields a concrete string from the current row, generalize the final code before promoting it into the real step implementation.

**Token discipline.** Always call `$CLI snapshot` with `--filename=.playwright-cli/step-<N>.yml`. That flag is what makes the CLI return just a file path instead of dumping the full a11y YAML inline to stdout ŌĆö which would be copied into the conversation and eat tokens for every snapshot. After writing the file, use `Read` with `offset` and `limit` to load only the region around the relevant element. Do not read the whole file into context. Re-read only when the page has materially changed.

Practical defaults:

- For `When` interaction steps: snapshot reuse first, then read-only DOM query only if needed, then hand-write the locator into `impl.js`
- For `Then` assertion steps: prefer writing the smallest clear assertion from the existing snapshot
- For container elements such as `generic`, `list`, `ul`, or broad `div` nodes: prefer finding a better child target from snapshot or a read-only DOM query rather than anchoring the final locator on the container itself
- For an immediate `Then` after a successful `When`, default to zero additional CLI calls; satisfy the assertion from the previous snapshot unless that snapshot genuinely lacks the assertion target
- When a locator keeps failing after 2+ rewrites despite snapshot analysis, or when a step times out, **take a screenshot** ŌĆö visual state often reveals what the accessibility tree cannot (overlay, clipping, spinner, unexpected modal). See `references/retry-policy.md` for the full screenshot trigger list.

Resolve the active `runId` from `latest-run.json`, then write inside that run directory:

```bash

```

It must be CommonJS and export an async function. `this` is the project's World object (type from `world.type` in the profile).

Write `impl.js` from the snapshot and read-only DOM evidence. Do not rely on MCP-generated action snippets.

**MANDATORY PRE-WRITE CHECKLIST:**

Before writing any code in impl.js, verify you have:

1. Ō£ģ Read `references/code-patterns.md` **in full** ŌĆö not just locator rules, but ALL coding patterns, page access conventions, assertion styles, and forbidden patterns
2. Ō£ģ Applied the testid scan from `references/cli-snapshot-rules.md` to the current snapshot
3. Ō£ģ Chosen the highest-priority locator available for the target element
4. Ō£ģ Verified the chosen approach (locator + page access + assertion) matches ALL patterns from Phase 0 project files and code-patterns.md

**If you cannot check all four boxes above, STOP and complete the missing steps before writing impl.js.**

**Choose the pattern that matches `page_access.pattern` from the profile:**

**Pattern: `world_property`** ŌĆö World has a `page` property directly:

```javascript
module.exports = async function () {
  await this.page.getByTestId('nav-media').click();
};
```

```javascript
module.exports = async function (optionType) {
  await this.page.getByRole('navigation')
    .getByRole('link', { name: optionType })
    .click();
};
```

For assertions:

```javascript
const { expect } = require('@playwright/test');

module.exports = async function () {
  await expect(this.page.getByRole('heading', { name: 'Latest News' })).toBeVisible();
};
```

**Pattern: `page_manager`** ŌĆö World has a PO manager (e.g., `this.pages`):

```javascript
module.exports = async function () {
  await this.page.getByTestId('nav-media').click();
};
```

Note: Even with a PO manager, `impl.js` uses `this.page` directly because the impl is a temporary bridge file ŌĆö the final PO delegation happens in Phase 4 when writing real step-defs and PO methods.

**Pattern: `static_singleton`** ŌĆö Page accessed via a static property (e.g., `BasePage.basePage`):

```javascript
const BasePage = require('../../src/pages/base.page').default;

module.exports = async function () {
  await BasePage.basePage.getByTestId('nav-media').click();
};
```

The `require` path must be relative from the `mcp-bridge/<runId>/` directory to the project source. Compute it based on the actual project layout.

#### E. Wait for result

```bash
node "$SKILL_DIR/scripts/wait-for-result.js" N --timeout-ms 60000
```

- `PASS` => proceed to next queued step
- error text => diagnose and rewrite `impl.js` ŌĆö the stub is still alive; just overwrite the file and it retries automatically (no restart needed)
- timeout => assume the implementation hung or the local waiter budget was too short; inspect the error, then retry or restart as appropriate
- `run_timeout` => the whole attach run expired; restart the run instead of rewriting the same `impl.js` blindly

#### F. Retry and failure handling

Read `references/retry-policy.md` for the full retry rules, failure categories, deadline management, blocked protocol, and run restart protocol.

Key points (details in the reference):

- Retry up to 6 times per step
- Diagnose before fixing ŌĆö state a hypothesis for *why* it failed
- If MCP confirms content is absent on the correct page, that is a **certain case error** ŌĆö stop immediately, do not burn retries
- Check `runDeadlineAt` before starting new retry loops (>= 90s: normal, < 90s: finish current only, < 30s: restart)
- After 6 failures, report BLOCKED and wait for user
- If a locator keeps failing after 2+ rewrites, or a step times out, take a screenshot via `$CLI screenshot` ŌĆö it often reveals what the accessibility snapshot cannot (overlay, spinner, unexpected modal, off-screen element). See `references/retry-policy.md` for the full trigger list.

**Normal retry (error text from `wait-for-result`):** just rewrite `impl.js` ŌĆö the stub process is still alive and will pick up the new file automatically. No restart needed.

**Run restart** is only required when: `run_timeout` occurs, `wait-for-result` times out (impl hung), or deadline < 30s. To restart: `stop-run.js --clean-port` then `start-run.js` (auto-restores validated impls).
